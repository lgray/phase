## 1. Measured ground truth and the soundness theorem

### 1.1 The LKI trace — which reads are FROZEN vs LIVE (the #4269 crux, proven not asserted)

Traced end-to-end (re-pinned @bceec86e3 — the quantity.rs/zones.rs/turns.rs production anchors below are grep-verified UNMOVED at tip; only the quantity.rs test-module lines shifted, noted below):

- **Freeze point:** `zones.rs:150–179` — on ANY battlefield/exile exit, an `LKISnapshot` is inserted into `state.lki_cache` capturing `power`, `toughness`, `base_power/toughness`, `card_types`, `counters` (:171), `tapped`, controller/owner — **at zone exit, before any dies-trigger is collected or resolved** (CR 400.7 look-back snapshot; CR 122.2 the live object's counters cease to exist on the move).
- **Read point:** `quantity.rs:3745–3796` `resolve_object_pt` — for `ObjectScope::Source` with a triggered ability (`source_incarnation.is_some()`, :3785), the read "expects the battlefield" and routes through `read_object_pt_by_id` (:3728–3743): live-battlefield value first, **LKI fallback when the source has left** (:3741). Doc comment :3759–3767 names Nested Shambler and CR 603.10a explicitly. CR 608.2h governs the read-at-resolution timing.
- **Pinning test:** `resolve_source_power_prefers_lki_when_source_left_battlefield` (quantity.rs:9715–9760) — buffed LKI (3/3) wins over reverted graveyard base (1/1) ⇒ token count 3. Sibling test :9767 proves live-battlefield wins while on-battlefield (the zone guard's other arm).
- **Persistence:** the LKI snapshot persists until step transition (quantity.rs:3770–3776 comment) — ordering and resolution of a departure batch happen within the step, so the snapshot is stable across the whole group's resolution.
- **Mutability by siblings (R1-corrected):** the SOLE production `lki_cache` writer is the zone-exit insert (`zones.rs:178`); the only other production touch is the step-transition `clear()` (`turns.rs:429`, CR 400.7 step-scoping); `log.rs:1325` is inside the test module (`#[cfg(test)]` :1239, `mod tests` :1240 — R2/m-3 pin) — test-only (log.rs's production use at :47 is a READ). **No trigger resolution mutates an existing LKI entry DIRECTLY** — but a sibling CAN mutate it indirectly, because `ObjectId` is stable across zone moves (`zones.rs:690` mutates `obj.zone` in place — the object is never re-keyed) and `zones.rs:178` is a same-id `HashMap::insert` (an OVERWRITE) fired on ANY battlefield/exile exit. Two indirect hazards: **(a) read goes live** — a sibling's membership write returns the departed object to the battlefield, and the zone-guarded read (quantity.rs:3728–3743) prefers the live object over the LKI; **(b) LKI overwritten** — the object exits battlefield/exile AGAIN within the window (re-death, or an exile round-trip), re-snapshotting different values under the same id. Both hazards require a group membership write that MOVES a member's existing object: (a) needs a battlefield-destination move; (b)'s re-death needs a prior battlefield re-entry (the overwrite fires at the re-death's battlefield exit, reachable only after (a)'s move) — **so re-death is SUBSUMED by the battlefield-destination condition**; (b)'s exile round-trip needs an out-of-exile move (exile ENTRY writes nothing; only exit does). A sibling that creates tokens, moves counters, or pumps survivors touches none of these — token/copy creation mints FRESH ObjectIds that cannot collide with a departed member's id. This is the mechanism behind the freeze-invalidation feed row (§1.2/D-profile): frozen reads conflict with external existing-object moves whose destination is the battlefield or whose origin is exile.

**Consequence (the frozen/live dichotomy, R1-corrected):** for a trigger whose source has departed, all Source-relative object reads are **frozen constants** bound at departure (CR 603.10a + CR 400.7) — **and they STAY frozen only while no group write re-enters the object onto the battlefield or moves it back out of battlefield/exile** (the two hazards above; ObjectId stability across zone moves, zones.rs:690, is what makes same-id re-binding possible at all). The gate therefore grants frozen treatment only when the group's write-set contains no freeze-invalidating move (§1.2). For a trigger whose source is on the battlefield, Source reads are **live** and sibling-mutable. Event-object reads on a `ZoneChanged`-from-battlefield event (the departed object's P/T/counters/controller, `HadCounters`, `ParentObjectTargetController`-style bindings) are frozen by the same snapshot. Live board-scoped reads (`ObjectCount`, `Aggregate`, graveyard counts, `HasCounters`-over-sets) are **never** frozen by departure.

### 1.2 The commutation theorem (the rules-correct gate predicate, CR 603.3b)

A group admitted by `group_is_order_independent` has: normalized-identical ASTs (`normalize_ability_identity` strips only `source_id`, :3377–3385), equal trigger-level `condition`/`subject_match_count`/`may_trigger_origin` **and equal `die_result` (R1/M1: `PendingTrigger.die_result`, triggers.rs:98, is stamped into `StackEntryKind::TriggeredAbility` at :4079/:4118 and read at resolution — CR 706.2 + CR 603.12; a pair differing only in it would NOT be the same state transformation, so the conjunct is added even though it is measured NOT constructible through the production ordering entry points today: every ordering-queue creation site stamps `die_result: None`, and the only `Some`-stamping sites — the reflexive-trigger pushes at effects/mod.rs:1584/:1683 and effects/choose_from_zone.rs:109 @bceec86e3 — bypass `begin_trigger_ordering` entirely via `push_pending_trigger_to_stack_with_event_batch`. Strictly conservative defense-in-depth: it can only shrink admitted groups; unit-pinned by N-F, §5.4)**, and no ordering input (:3504–3512, CR 603.3c/603.3d — targets/modes/divisions announced at placement; CR 603.5 "may"/unless-pay choices are resolution-time and excluded). Each member's **resolution function** is therefore the same program `f` parameterized only by its two live bindings: `(source, event)`. Ordering is immaterial iff the composed functions commute pairwise. Grounds:

- **T1 (identical functions, R1-completed):** each member's resolution function is the shared program `f` bound to its `(source, event)` pair — so members are literally the same state transformation whenever the bindings are equal OR the program never consults the differing binding. Concretely: **(events equal OR every event-context read is frozen) AND (all members share one `source_id` OR the profile is source-independent — `reads_src = ∅ ∧ writes_self = ∅ ∧ no source-referential condition predicate`)** ⇒ every permutation yields the same composition `f∘…∘f`. Sound **modulo the source-actor residual (side condition below — R2/B-NEW-1)**. The source-independence disjunct is MEASUREMENT-FORCED, not optional: the §1.3.1-E class combines a board-count read with a membership write (**57 TRIGGERS = 56 unique cards** — printed non-legendary same-event no-input, pinned effect-keyed def; Keeper of the Accord carries two qualifying triggers (R3/F2 unit pin); Endless Ranks of the Dead class), of which the **38 source-independent TRIGGERS (= 37 unique cards)** are provably order-independent by this lemma but would falsely conflict under the same-kind `SetMembership` feed row and flip auto→prompt without it (the other 19 are the named source-dependent rows of §1.3.1-E/F). Scoped to `same_event` on the gate (§1.2 fast path); the batch path keeps the full conflict machinery (frozen-constant bindings differ per member, so source-independence alone does not make batch members identical).
- **T2 (frozen bindings are constants, R1-corrected):** a read proven frozen (§1.1) is a per-member constant **for as long as the freeze is valid** — i.e. while the departed object neither re-enters the battlefield (read goes live: the zone-guarded read at quantity.rs:3728–3743 prefers live-battlefield state, and ObjectId is stable across zone moves, zones.rs:690) nor re-exits battlefield/exile (same-id `lki_cache.insert` overwrite, zones.rs:178). Under a valid freeze it cannot observe sibling resolutions; functions differing only in frozen constants commute **iff their writes pairwise commute**. Freeze validity is enforced by the freeze-invalidation feed row (below): frozen treatment is granted only when the group's write-set contains no external existing-object move with battlefield destination or exile origin (re-death overwrite subsumed — §1.1).
- **T3 (write commutation, fail-closed):** writes commute when they are (a) disjoint — Source-scoped writes of distinct sources ("put a counter on ~", "return ~ to hand"), or (b) identical order-insensitive accumulations from the shared program (counter/life/token deltas — same delta applied twice). What breaks commutation is exactly a **read/write feed**: one member's write lands in a location class another member's *live* read observes (PR-6.25 §1 Case A: live `Power{Source}` threshold read × board-wide counter write; the representable batch counterexample: live graveyard-count read × return-self-to-battlefield membership write). **R2/B-NEW-1: (b)'s "same delta" holds only modulo the source-actor side condition (below)** — per-source granted state (lifelink/deathtouch class) modulates the delta's resolution without appearing in the normalized AST, and a player-eliminating `PlayerLife` delta cascades into object removal (CR 800.4a). The side condition applies to the BATCH path too: read-free damage dies-triggers form departure batches (two identical "when this dies, deal N damage to each opponent" sources, one lifelink-granted, commute only modulo the residual).

1. **Granted-state modulation (CR 702.15 lifelink / CR 702.2 deathtouch class):** an aura/counter/effect granting lifelink to ONE copy leaves both normalized ASTs identical (normalization strips only `source_id`; granted state lives on the source object, not the AST), so the group forms — but damage dealt by the granted member also gains its controller life at resolution: the members are no longer the same function, and the read-based conflict formula cannot see it (the modulator is not a read).   let sibling_writes_for_src_reads   = profile.writes_external                            // distinct sources: self-writes are disjoint from MY source
```
                                        ∪ (structure.all_same_source ? profile.writes_self : ∅)
  let sibling_writes_for_board_reads = profile.writes_external ∪ profile.writes_self      // board reads see the sibling's own source too

      && (profile.reads_frozen ≠ ∅ ∨ profile.reads_src ≠ ∅ ∨ profile.reads_event_live))  // freeze-invalidation row: hazard write ×
                                                                                          // would-be-frozen reads ⇒ conflict (B1)
  ∨ feeds(live_src_reads,  sibling_writes_for_src_reads)
  ∨ feeds(profile.reads_board ∪ profile.reads_player_mutable, sibling_writes_for_board_reads)
```

### 1.4 Why 1-bit read-precision CANNOT deliver the parity bar (the decisive counterexamples)

```rust
/// Location-kind lattice for CR 603.3b sibling-conflict analysis. Closed set;
/// every kind names the game state it stands for and the CR-visible writes that feed it.
enum StateKind { ObjectPt, ObjectCounters, SetMembership, PlayerLife, HandLibrary,
                 JournalLife, JournalCards, JournalCast, StackShape, TapState, Other /* always-conflict */ }
struct KindSet(/* bool per StateKind — mirror Axes' explicit-bool style, no bitmagic */);
struct RwProfile {
    reads_src: KindSet,          // Source-scoped reads ONLY (live unless structure freezes them).
                                 // R1/M4: Recipient-scoped reads are NOT recorded here — a Recipient
                                 // read is the write's own modify-input (read-modify-write): it observes
                                 // the object the member is currently writing, and identical members'
                                 // composed per-object totals are symmetric (each object is touched by
                                 // the same map the same number of times; the self-exclusion asymmetry
                                 // cancels — Canopy Gargantuan proof, §1.3.1-G). N-E pins the pair:
                                 // Recipient-shape clean vs Source-shape (Ouroboroid) conflict.
    reads_board: KindSet,        // board/graveyard/stack-top-scoped mutable reads
    reads_player: KindSet,       // life totals, hand size, journals (sibling-mutable player state)
    reads_frozen: KindSet,       // CR 603.10a look-back class: HadCounters, event-object reads
                                 // (never conflict WHILE THE FREEZE IS VALID — see writes_reentry_hazard)
    reads_event_live: bool,      // event reads NOT proven frozen — only consulted by T1's fast path
    legacy_batch_prompt: bool,   // D5: the 12 retained-prompt refs
    writes_self: KindSet,        // writes scoped to own Source/Recipient
    writes_external: KindSet,    // board-, player-, stack-scoped writes
    writes_reentry_hazard: bool, // R1/B1: an EXTERNAL move of EXISTING objects whose destination is the
                                 // battlefield or whose origin is exile (ChangeZone/ChangeZoneAll-class
                                 // payloads; token/copy/conjure creation exempt — fresh ObjectIds cannot
                                 // collide with a departed member's id, zones.rs:690; SelfRef /
                                 // TriggeringSource-class self-returns exempt — a member's own-source
                                 // move cannot precede its own read, and sibling ids are distinct;
                                 // library-origin moves exempt — cannot touch the batch's departed
                                 // objects; unknown/null origin ⇒ fail-closed hazard). Feeds the
                                 // freeze-invalidation row (§1.2). Parity: §1.3.1-I, zero printed diffs.

impl RwProfile {
    // R1/M4 (T1 completion): true iff no reads_src, no writes_self, and no
    // source-referential condition predicate (the walk routes those into reads_src).
```

`feeds(reads, writes)`: same-kind intersection PLUS the cross-kind feed rows: `ObjectCounters → ObjectPt` (counters change P/T), `SetMembership → {ObjectPt, ObjectCounters}` only for *aggregate* reads (an `Aggregate`/`CountersOnObjects` read is recorded as its value-kind AND SetMembership, so membership writes conflict with it; a bare `ObjectCount` is SetMembership only), `PlayerLife → JournalLife`, `JournalCards ← HandLibrary` draw/discard/mill writes, `StackShape → StackShape`, **and the freeze-invalidation row (R1/B1): `writes_reentry_hazard × (would-be-frozen reads on a departure batch)` — a direct conjunct, not a promotion (§1.2)**. **Census-overlap refinement (R1/M4, forced by §1.3.1-F):** the `SetMembership` same-kind row fires only when the write's moved-object census can OVERLAP the read filter's requirements — reads carry their filter's extractable requirements (core types, subtypes, structured `FilterProp`s like `Token`, controller axis); writes carry their moved census (SelfRef moves ⇒ the group's live source census, supplied by `GroupStructure`; filter-scoped moves ⇒ the filter's census; creation ⇒ the created token's typeline + controller); ANY unextractable component on either side ⇒ overlap assumed (fail-closed).
