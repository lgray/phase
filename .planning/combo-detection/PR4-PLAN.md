## 1. Objective + explicit NON-goals

### Objective
Add `analysis::ability_graph` ŌĆö an **offline, static** extractor that, given a list of card faces (a deck / a claimed combo's cards), builds a directed ability/resource graph, finds strongly-connected components (Tarjan SCC), and emits **candidate cycles** whose summed per-cycle resource vector is *coverable* (net-progress: Ōēź1 axis strictly up, no controller-side consumed axis net-negative). Each candidate names the unbounded `ResourceAxis` family it would pump, so PR-5's confirmer (Engine A `detect_loop`) can be fed a card list and a set of expected axes.

This is **Engine B** from `FEASIBILITY-AND-PLAN.md:83-88`: the over-approximate, fast, card-list stage that *proposes* candidates; Engine A (`detect_loop`, already shipped) is the sound, stateful stage that *confirms* them. Either runs standalone.

### NON-goals (explicit ŌĆö these are PR-5..8, not PR-4)
- **No resolver / stack / priority / SBA / reducer changes.** PR-4 never calls `apply()`, never drives a `GameRunner`, never touches `game/engine.rs`, `game/stack.rs`, `game/sba.rs`, `loop_detect_ring`, or `reconcile_terminal_result`. Those are PR-3's live machinery and must not be perturbed.
- **No `GameState` involvement at all.** Engine B works purely from `&CardFace` ASTs. (Engine A is the GameState path; they share only the `ResourceVector`/`ResourceAxis`/`WinKind` vocabulary.)
- **No CLI binary, no `.cargo/config.toml` alias.** `cargo combo-verify` is PR-5. PR-4 adds library surface only.
- **No multiplayer-filter, frontend, WASM-bridge, or AI (phase-ai) wiring.** PR-8 maps the engine `WinKind` onto `phase-ai`'s `combo::WinKind`; PR-4 must NOT import phase-ai (that would invert the dependency edge ŌĆö phase-ai depends on engine).
- **No confirmation / soundness claim.** A candidate is *unconfirmed by construction* (over-approximate: it ignores targeting legality, timing windows, "may" choices, replacement interactions). PR-4 must never label a candidate a `LoopCertificate` (see ┬¦3, output type).

**This PR is purely additive and changes ZERO game behavior.** It is verified the same way the existing `analysis` module is: inline `#[cfg(test)]` unit tests + the gitignored-export-gated corpus smoke test. ### 3.2 The per-`Effect` projection ŌĆö `Projection`

```rust
/// Per-axis production magnitude marker (HIGH-1). A production axis is either a
/// fixed integer amount or unbounded-up (a dynamic `QuantityExpr` ŌĆö
/// `Ref`/`Multiply`/`UpTo` ŌĆö sitting in a PRODUCTION position). Dynamic
/// production must NOT be under-counted to a unit, or recall-critical rows
/// (Priest of Titania + Umbral Mantle, net +1..+3 mana) collapse to a net
/// deficit and get dropped by coverability.
enum AxisMagnitude {
    Fixed(i32),
    /// Unbounded-up: dynamic production. Coverability treats this axis as
    /// coverable regardless of any fixed counter-cost on the same axis.
    Unbounded,
}

/// The static resource contribution of a single Effect node. Typed enum (NOT a
/// `modeled: bool`, LOW-1) so the "unmodeled contributes nothing" invariant is
/// unforgeable. Over-approximate by design (candidate stage).
enum Projection {
    /// This Effect variant has a concrete mapping. Carries the signed net
    /// `ResourceVector` (positive = produced/gained, negative = consumed/cost,
    /// `ResourceVector` axes verbatim) PLUS per-produced-axis magnitude markers
    /// so dynamic production records `Unbounded`-up (HIGH-1).
    Modeled {
        vector: ResourceVector,
        /// Keyed by the produced `AxisKey`. `Fixed(n)` for static amounts;
        /// `Unbounded` for dynamic `QuantityExpr` in a production position.
        magnitudes: BTreeMap<AxisKey, AxisMagnitude>,
    },
    /// No concrete mapping: `Unimplemented`, or a variant deferred to PR-4b.
    /// Contributes nothing (default vector, no magnitudes, no edges).
    Unmodeled,
}
```

`fn effect_projection(effect: &Effect) -> Projection` ŌĆö **exhaustive `match` over all 207 `Effect` variants** (`types/ability.rs:7767-10491`), no wildcard. Modeled families in **this PR** (the 5 the prompt names ŌĆö ~58 variants, measured in MAP-2):

- **mana** (`Mana` `:8895`, `GainEnergy`, `ReduceNextSpellCost`, ŌĆ”): `Mana.produced: ManaProduction` (`:1252`) ŌåÆ positive on `vector.mana[color_index]` (the per-color `ResourceVector` field is retained for magnitude bookkeeping), but the **edge-matching key collapses to the single color-agnostic `AxisKey::Mana`** (R3-MANA-COLLAPSE, ┬¦3.4/┬¦3.5): the magnitude marker is keyed `magnitudes[AxisKey::Mana]`, and the color-flexible `ManaProduction` variants (`AnyOneColor`, `AnyCombination`, `ChosenColor`, `OpponentLandColors`, `AnyTypeProduceableBy`, `ChoiceAmongExiledColors`, ŌĆ”) all fold to that one key **for edge matching**.
  - **TWO-LAYER SEPARATION (R4-MED1-COLOR-SEED ŌĆö explicit, removes the ┬¦3.2-vs-┬¦3.7/┬¦6 D contradiction).** The "cannot be pinned to a single `ManaType` / all fold to that one key" statement is scoped to the **EDGE-KEY layer ONLY**: every mana production, whatever its color, collapses to the single `AxisKey::Mana` so producer/consumer edges intersect (R3-MANA-COLLAPSE). **VECTOR SEEDING is a SEPARATE layer.** The per-color `ResourceVector.mana[color_index]` slot ŌĆö which ultimately feeds the candidate's output `unbounded: Vec<ResourceAxis>` (┬¦3.7 step 5) ŌĆö DOES read a determinate `ManaType` whenever the production pins **exactly one statically-knowable color**: `Fixed` and `Mixed` (their single determinate colored element) and `AnyOneColor`/`AnyCombination` **with a singleton `color_options` set** all seed that determinate color slot. Only genuinely multi-color-or-unknowable productions seed the `Colorless` sentinel slot: multi-element `color_options`, `ChosenColor` (a *previously-/runtime-chosen* color ŌĆö it carries **no** static `color_options` field at `ability.rs:1295-1309`, so its color is undeterminable at static time), `OpponentLandColors`, `AnyTypeProduceableBy`, and `ChoiceAmongExiledColors`. **Priest of Titania's measured AST is `AnyOneColor { color_options: [Green] }` (singleton, `ability.rs:1275-1286` confirms `AnyOneColor` carries both `count: QuantityExpr` for the per-Elf dynamic amount and `color_options: Vec<ManaColor>`) ŌåÆ seeds `Mana(Green)`**: the edge still matches color-agnostically (Priest's green production Ōł® Umbral's generic `{3}` cost both carry `AxisKey::Mana`) while the seeded vector retains real green, making ┬¦3.7 step 5 / ┬¦6 D's "`unbounded` names `Mana(Green)`" internally consistent ŌĆö the edge-key fold and the vector seed are two different layers, and an executor must seed `Mana(Green)` (not the `Colorless` sentinel) for Priest. `GainEnergy` ŌåÆ `(Energy, Player)` counter axis.
- **counters** (`PutCounter` `:8739`, `PutCounterAll`, `RemoveCounter` `:8000`, `Proliferate`, `MultiplyCounter`, ŌĆ”): `counter_type` ŌåÆ `CounterClass::from_counter_type` ŌåÆ `(class, ObjectClass::*)`; sign from put (+) vs remove (ŌłÆ); `Proliferate` ŌåÆ `generic_triggers[Proliferate] += 1`. - **damage** (`DealDamage` `:7786`, `DamageAll`, `DamageEachPlayer`, `EachDealsDamageEqualToPower` `:7826`, ŌĆ”): ŌåÆ `damage_dealt[OPPONENT] += magnitude` (see ┬¦3.6 player convention).
- **untap/tap** (`SetTapState` `:7989`, field `state: TapStateChange` `:3736`, variants `Untap`/`Tap`): match on `TapStateChange` for BOTH polarities (LOW-2) ŌĆö `Untap` **produces** untapped state (`AxisKey::Tap` as a producible resource ŌĆö the loop pivot for mana untap engines), `Tap` **consumes** untapped state (Opposition-style engines) ŌåÆ `AxisKey::Tap` in `requires`. Represented via the node `produces`/`requires` axis-sets, not a `ResourceVector` numeric axis (`ResourceVector` has no tap axis ŌĆö ┬¦3.5). **NOTE (HIGH-1 RESIDUAL):** an untap in the *cost* position (`AbilityCost::Untap`, the `{Q}` symbol ŌĆö `ability.rs:6445`) is the **symmetric** producer of `AxisKey::Tap` and is handled in the ┬¦3.3 cost fold / ┬¦3.5 edge model, **not** here (this `effect_projection` arm only covers EFFECT-position `SetTapState`). The `{Q}`-untap-cost class (Umbral Mantle, Sword of the Paruns, Staff of Domination, Magewright's Stone, Kiora's Follower) closes its loop entirely through the cost-position rule.
- **cast/copy** (`CopySpell`, `CastFromZone`, `Cascade`, `Conjure`, `Myriad`, ŌĆ” MAP-2's 24): ŌåÆ `casts_this_step += 1` and/or `tokens_created` for copy-to-token; copy/cascade that recasts ŌåÆ cast axis.

Magnitude rule (HIGH-1 ŌĆö per-axis, polarity-aware; this replaces the old uniform unit-magnitude model):
- **Production positions** (positive axes): extract `QuantityExpr::Fixed { value }` (`:4904`) ŌåÆ `AxisMagnitude::Fixed(value)`. For dynamic `Ref`/`Multiply`/`UpTo`/etc. ŌåÆ `AxisMagnitude::Unbounded` (unbounded-up) for that axis, seeding `vector` with unit so the axis is present/positive. **Never under-count dynamic production to a unit** ŌĆö that is the bug that drops Priest of Titania + Umbral Mantle (nets +1..+3 mana, which a unit model scores as ŌłÆ2 mana = rejected by coverability).
- **Cost positions** (negative axes, from `AbilityCost` in ┬¦3.3): dynamic cost amounts stay at **unit (1)** ŌĆö a recall-safe *under-approximation* of cost (under-counting a cost can only make a cycle look more coverable, never less, so it cannot cause a false-negative miss).

This keeps the static stage recall-focused: a candidate the confirmer later rejects is acceptable; a false-negative miss is not. Finer fixed-magnitude precision on the cost side is a PR-4b refinement; the production `Unbounded` marker is required in PR-4a.

All remaining ~149 variants (P/T, control, combat-status, dice, win/lose, zone-change, draw/mill, life, tokens, search, structural, **and `Effect::Unimplemented` `:10486`**) ŌåÆ `Projection::Unmodeled`. The `Unmodeled` arm is the graph-layer analogue of `Effect::Unimplemented` and is treated as contributing nothing (a node consisting only of unmodeled effects is a graph sink/source with no resource edges).

> **Why exhaustive-match, not the serde `"type"` tag** (`effect_type_name`, `coverage.rs:415`): the FeatureSupport precedent (`coverage.rs:5908`) is explicit ŌĆö an exhaustive `match` makes a newly-added `Effect` variant a **compile error** until classified, preventing the silent drift a string-tag dispatch allows. This is mandatory per the design principles and the prompt's section 4.

### 3.3 Node model ŌĆö `AbilityNode`

One node per *ability* across the input faces:
- spell/activated `AbilityDefinition`s (`face.abilities`),
- each `TriggerDefinition.execute` (`face.triggers`) ŌĆö `execute: Option<Box<AbilityDefinition>>` (`:15492`); **a trigger whose `execute == None` produces no node** (LOW-4: the walker/node-builder skips `None`),
- each `ReplacementDefinition.execute` (`face.replacements`) ŌĆö `execute: Option<Box<AbilityDefinition>>` (`:16115`); **a replacement whose `execute == None` produces no node** (LOW-4),
- `ContinuousModification::GrantTrigger`/`GrantAbility` children of statics (mirroring `build_parse_details` `:3784-3794`).

```rust
struct AbilityNode {
    face_name: String,                       // provenance: which card this ability came from
    kind: AbilityKind,                       // Spell / Activated / ... (types/ability.rs:12852)
    net: ResourceVector,                     // folded Projection vectors over collect_effects + cost vector
    unbounded_production: BTreeSet<AxisKey>,  // HIGH-1: axes produced with AxisMagnitude::Unbounded
    produces: BTreeSet<AxisKey>,             // axes this node drives strictly UP (incl. tap-untap, events it emits)
    requires: BTreeSet<AxisKey>,             // axes this node costs/needs + the trigger-event axis that fires it
    any_unmodeled: bool,                     // Ōēź1 collected effect was Unmodeled (candidate confidence flag)
}
```

`net` folds, over every `e` from `collect_effects` over `def.effect`+chains: (a) for each `Projection::Modeled { vector, magnitudes }`, add `vector` into `net` and union each `Unbounded` axis from `magnitudes` into `unbounded_production` (Fixed amounts are already carried in `vector`); `Projection::Unmodeled` contributes nothing and sets `any_unmodeled = true`; and (b) the **cost**, folded by an **exhaustive no-wildcard `match` over all 29 `AbilityCost` variants** (`Tap`..`Unimplemented`, `ability.rs:6431-6624` ŌĆö fourth drift gate, ┬¦4). A cost contributes to `net` as a **negative** component for resources it *consumes* and as a **positive** component for resources it *produces in cost position* (so the field-bearing axes flow through the `From<&ResourceAxis>` derivation of ┬¦3.4 automatically); field-less axes (`Tap`, `AnyCounter`) are injected directly into `produces`/`requires`. The classified arms:

  - `Mana` / `ManaDynamic` (`:6444`-adjacent) ŌåÆ negative `mana[]` (dynamic cost amounts at unit, HIGH-1). Under R3-MANA-COLLAPSE this surfaces as `requires:AxisKey::Mana` (any negative `mana[]` color folds to the one `Mana` key via `From<&ResourceAxis>`, ┬¦3.4) ŌĆö a colored cost requires the single fungible `Mana` axis.
  - **`PayLife` (`:6455`) ŌåÆ R3-LIFE-SYMMETRY: NO life-axis contribution in PR-4a (no-op on the life axis).** Keying only the cost side while `Effect::GainLife`/`LoseLife` (`:7960`/`:7971`) stay unmodeled would make `AxisKey::Life` appear only in `requires`, never `produces` ŌĆö so no life edge could ever form, AND the resulting controller-side net-negative life would trip candidate coverability (controller-side consumed axis net-negative & not unbounded ŌćÆ reject), dropping *any* otherwise-valid (e.g. unbounded-mana) cycle that incidentally pays life. The recall-safe resolution is to defer **both** life positions together to PR-4b (model `PayLife` cost *and* `GainLife`/`LoseLife` effects in the same slice); under-approximating a pay-life *cost* (ignoring it) can only make a cycle look more coverable, never less, so it causes no false-negative.
  - `PayEnergy` (`:6522`) ŌåÆ negative `counters[(Energy, Player)]` ŌćÆ `requires` (energy is single-type and symmetric with `Effect::GainEnergy`, no fungibility issue).
  - `RemoveCounter { counter_type }` (`:6514`) ŌåÆ **R3-COUNTER-FUNGIBILITY:** match on `counter_type: CounterMatch` (`counter.rs:270`) ŌĆö `CounterMatch::OfType(t)` ŌåÆ negative `counters[(from_counter_type(t), obj)]` ŌćÆ `requires:Counter(class,obj)`; `CounterMatch::Any` (`counter.rs:272`, "remove a counter of ANY type") ŌåÆ inject the requires-only wildcard **`AxisKey::AnyCounter`** (it cannot be pinned to any one `CounterClass`), which the edge model (┬¦3.5) intersects against **any** `Counter(_, _)` producer.
  - **`Loyalty { amount: i32 }` (`:6446`) ŌåÆ R3-COUNTER-COST-SYMMETRY (sign-aware, bidirectional):** a planeswalker `[+N]` ability *adds* loyalty counters as its cost, so `amount > 0` ŌåÆ positive `counters[(Loyalty, Planeswalker)]` ŌćÆ `produces:Counter(Loyalty,Planeswalker)`; `amount < 0` (a `[ŌłÆN]` ability) ŌåÆ negative ŌćÆ `requires`; `amount == 0` ŌåÆ no-op.   - **`Blight { count: u32 }` (`:6550`) ŌåÆ R3-COUNTER-COST-SYMMETRY:** puts `count` ŌłÆ1/ŌłÆ1 counters as a cost ŌćÆ positive `counters[(Minus1Minus1, Creature)]` ŌćÆ `produces:Counter(Minus1Minus1,Creature)`.

### 3.4 `AxisKey` ŌĆö the edge-matching key

`ResourceAxis` (`resource.rs:482`, **16 variants**) carries concrete `PlayerId`/magnitude, unsuitable as a *relational* edge key. Introduce a small `AxisKey` enum in `ability_graph.rs` = the *family* of an axis without magnitude/player-id (`Mana` (R3-MANA-COLLAPSE ŌĆö a **SINGLE color-agnostic family**, NOT `Mana(ManaType)`; see below), `Damage`, `Life`, `Library`, `Counter(CounterClass, ObjectClass)`, `AnyCounter` (R3-COUNTER-FUNGIBILITY ŌĆö a **requires-only wildcard** for `CounterMatch::Any` costs; ┬¦3.3/┬¦3.5), `Trigger(TriggerKind)`, `Tokens`, `Casts`, `Draw`, `Tap`, `ExtraTurn`, `Etb`, `Ltb`, `Death`, `Sac`, `Landfall` (R3-LANDFALL-COLLAPSE ŌĆö the **SINGLE** landfall key; both `ResourceAxis::LandfallTriggers` and `ResourceAxis::Trigger(TriggerKind::Landfall)` fold here so the two representations of one resource always intersect), `Combat`). This is a *leaf parameterization* of the same axes ŌĆö derived from `ResourceAxis` by dropping the runtime payload ŌĆö so it does not violate the categorical-boundary rule (same CR sections; it is the static-edge projection of the existing axis vocabulary, one abstraction level up in genericity). Document it as "the magnitude/player-agnostic projection of `ResourceAxis` used only for static edge matching."

**R3-MANA-COLLAPSE (lead decision, review round 3).** `AxisKey::Mana` is a **single color-agnostic family** ŌĆö there is no per-color `Mana(ManaType)` key. Any mana production (any color OR colorless) ŌåÆ `produces:AxisKey::Mana`; any mana cost (generic, colored, OR hybrid) ŌåÆ `requires:AxisKey::Mana`. This is a **deliberate recall-first over-approximation**: a colored cost (e.g. `{G}`) satisfied by off-color production (e.g. a `Mana(Red)` source) is a **known recall-safe false-positive**, because the static stage is intentionally over-approximate and every candidate is re-checked by Engine-A's *precise* floating-pool color accounting downstream (the existing runtime confirmer already does exact-color matching ŌĆö corroborated by `drive_combo_10_priest_umbral` at `corpus_tests.rs:1632`, which drives real colored mana through the reducer). The alternative ŌĆö keying mana per `ManaType` ŌĆö is **strictly wrong** for recall: (1) a generic `{3}` cost (`ManaCost::Cost { shards: [], generic: 3 }`, `mana.rs:1236-1267`; generic has no color slot, keyed `Colorless`) would fail to intersect a `Mana(Green)` production, and (2) the color-flexible `ManaProduction` variants (`AnyOneColor`, `AnyCombination`, `ChosenColor`, `OpponentLandColors`, `AnyTypeProduceableBy`, `ChoiceAmongExiledColors`) cannot be keyed to any single `ManaType` at all. Per-color precision is **deferred to PR-4b** if the measured false-positive rate warrants it (it is a leaf refinement of the edge intersection, not an architectural change). This collapse is enforced at compile time by the exhaustive `From<&ResourceAxis>` fold below (every `Mana(_)` arm maps to the one `AxisKey::Mana`).

**HIGH-2 ŌĆö compile-time drift gate from `ResourceVector`/`ResourceAxis` (binding executor requirement).** `AxisKey` shadows `ResourceAxis` (`resource.rs:482`, 16 variants) by dropping `PlayerId`/magnitude and *adding* the axes that have no `ResourceVector` field (`Tap`). A future `ResourceAxis`/`ResourceVector` axis (e.g. `Energy`) must not silently fail to appear in edge-matching. Therefore the `ResourceAxis ŌåÆ AxisKey` projection MUST be an **exhaustive no-wildcard `impl From<&ResourceAxis> for AxisKey`** ŌĆö every one of the 16 `ResourceAxis` variants mapped explicitly, **no `_ =>` arm** ŌĆö so a newly-added `ResourceAxis` variant is a **compile error until mapped**, identical to the `effect_projection`/`FeatureSupport` drift gate (precedent: `FeatureSupport` exhaustive match, `coverage.rs:5908`). Two folds are baked into this impl as the compile-time enforcement of the round-3 collapses: (1) **R3-MANA-COLLAPSE** ŌĆö `ResourceAxis::Mana(_)` maps to the single `AxisKey::Mana` for *every* `ManaType` (the `ManaType` payload is discarded); (2) **R3-LANDFALL-COLLAPSE** ŌĆö both `ResourceAxis::LandfallTriggers` **and** `ResourceAxis::Trigger(TriggerKind::Landfall)` map to the single `AxisKey::Landfall` (the `Trigger(kind)` arm matches on `kind`: `TriggerKind::Landfall ŌåÆ AxisKey::Landfall`, every other `kind ŌåÆ AxisKey::Trigger(kind)`), so the field-axis and the trigger-kind axis for the one landfall resource always intersect. `AxisKey::AnyCounter` is **never produced** by this `From` (it has no `ResourceVector` field); it is a requires-only sentinel injected solely by the `RemoveCounter`/`CounterMatch::Any` cost-fold arm (┬¦3.3).

**Computing `produces`/`requires` from the node's net `ResourceVector` (HIGH-2):** the derivation is a per-field exhaustive walk of `ResourceVector` (driven by / kept honest by `From<&ResourceAxis>`): each axis with a **strictly positive** net component (or present in `unbounded_production`, ┬¦3.3/HIGH-1) contributes its `AxisKey` to `produces`; each axis with a **strictly negative** net component contributes its `AxisKey` to `requires`. Because the ┬¦3.3 cost fold writes cost-position productions back into `net` as **positive** field components (`Loyalty[+N]`ŌåÆ`counters[(Loyalty,Planeswalker)]`, `Blight`ŌåÆ`counters[(Minus1Minus1,Creature)]`, `Sacrifice`ŌåÆ`sac_triggers`/`ltb_triggers`/`death_triggers`) and resource costs as **negative** components, those axes flow through this same walk automatically ŌĆö no extra injection needed. Only the `AxisKey`s that have **no `ResourceVector` field** are injected directly, at exactly three sites: (1) **`Tap`** from `SetTapState` (both polarities, LOW-2) / `AbilityCost::Tap` & `AbilityCost::TapCreatures` (ŌåÆ`requires`, R3-TAP-COST) / `AbilityCost::Untap` (ŌåÆ`produces`, the `{Q}` symbol; ┬¦3.5/HIGH-1 RESIDUAL); (2) **`AnyCounter`** from the `AbilityCost::RemoveCounter`/`CounterMatch::Any` arm (ŌåÆ`requires`, R3-COUNTER-FUNGIBILITY); (3) the **trigger-event axis** from `trigger_axis(&def.mode)` for `TriggerDefinition` nodes (┬¦3.5/MEDIUM-1). The `ResourceVector`-field walk plus those three injection sites are the *complete* set of `produces`/`requires` sources ŌĆö adding a `ResourceVector` field without extending the walk is a compile error via the `From<&ResourceAxis>` exhaustiveness.

### 3.5 Edge model ŌĆö resource producer/consumer

`AbilityGraph` is a **type alias** `type AbilityGraph = DiGraph<AbilityNode, ResourceEdge>` (LOW-3 ŌĆö not a newtype; it names the concrete petgraph type so the ┬¦7 re-export is not dangling). `fn build_ability_graph(nodes: Vec<AbilityNode>) -> AbilityGraph`:
- For ordered node pair (A, B), add edge **A ŌåÆ B** iff `A.produces` intersects `B.requires` (A produces an axis B needs/triggers on). Intersection is computed by an `axis_matches(produced, required) -> bool` helper, **not** raw set equality, so the two round-3 wildcard/collapse rules are honored: (i) **R3-COUNTER-FUNGIBILITY** ŌĆö `required == AxisKey::AnyCounter` matches **any** `produced == AxisKey::Counter(_, _)` (a "remove a counter of any type" cost is satisfied by any counter producer); plain `Counter(class,obj)` requirements still match only the equal `Counter(class,obj)`. (`AnyCounter` is requires-only, so the reverse direction never arises.) (ii) **R3-MANA-COLLAPSE / R3-LANDFALL-COLLAPSE** need no special casing here because the collapse already happened in `From<&ResourceAxis>` (┬¦3.4) ŌĆö both sides carry the single `AxisKey::Mana` / `AxisKey::Landfall`, so plain equality suffices. The edge stores the shared `AxisKey`(s) (`ResourceEdge { via: Vec<AxisKey> }`).
- Self-edges allowed (a single ability that both produces and requires the same axis ŌĆö e.g. Devoted Druid untap class is a self-loop once its `requires` is satisfied by an enabler node).

The `produces`/`requires` sets are how trigger-event and tap-untap edges are expressed without a numeric axis:
- a node whose `def` is a `TriggerDefinition.execute` puts its **trigger-event axis** in `requires` (e.g. a "whenever you gain life" trigger requires `Life`-gain; "whenever a creature dies" requires `Death`; "whenever you cast" requires `Casts`). This `TriggerMode ŌåÆ AxisKey` mapping is `trigger_axis(&TriggerMode) -> Option<AxisKey>`, a **second exhaustive no-wildcard classifier** that MUST match all **169** `TriggerMode` variants (`types/triggers.rs:210`) with no `_ =>` arm ŌĆö exactly like `effect_projection`'s `Effect` gate. In PR-4a the 4a Some-set is exactly **{cast, counter, tap, mana}** ŌĆö these trigger CONSUMERS return `Some(..)` because their matching PRODUCERS are modeled in 4a (cast/copy effects, `PutCounter`, `SetTapState` + `AbilityCost::Tap`/`Untap`, `Effect::Mana`); every other variant ŌĆö **including the `lifegain`/`dies`/`ETB`/`sac` trigger families** ŌĆö returns `None` **as an explicit arm**, so a future combo-relevant `TriggerMode` is a compile error until classified, never a silent false-negative. **Binding executor requirement (MEDIUM-1, lead decision ŌĆö NOT executor discretion); precedent `FeatureSupport`, `coverage.rs:5908`.** Broadening arms from `None` to `Some(..)` ŌĆö in particular `lifegain`/`dies`/`ETB`/`sac` ŌĆö is the main PR-4b lever (see ┬¦8, ┬¦6 B, ┬¦6 D, which already defer those consumers to 4b). **(R4-MED2-TRIGGER-4A-SET) Interaction with R3-SAC-POLARITY:** even though `AbilityCost::Sacrifice` PRODUCES `Sac`/`Ltb`/`Death` events in 4a (┬¦3.3), the dies/sac trigger CONSUMER side (`trigger_axis` for those modes) returns `None` until 4b ŌĆö so the **sac-costŌåÆdies-trigger edge does not close until 4b**. This is intended: 4a bakes in the producer polarity so 4b only flips the consumer arm, and it removes the apparent ┬¦3.5-vs-┬¦8/┬¦6 B contradiction (4a producers can exist without their 4a consumers).
- a `SetTapState` node with `state: TapStateChange::Untap` puts `AxisKey::Tap` in `produces` (produces untapped state); `state: TapStateChange::Tap` puts `AxisKey::Tap` in `requires` (consumes untapped state, Opposition-style); an activated ability whose cost is `AbilityCost::Tap` also puts `AxisKey::Tap` in `requires` (LOW-2 ŌĆö match on `TapStateChange` handles both polarities explicitly).
- **(HIGH-1 RESIDUAL ŌĆö the untap-COST edge)** an activated/granted ability whose cost is `AbilityCost::Untap` (the `{Q}` symbol, `ability.rs:6445`) puts `AxisKey::Tap` in `produces` ŌĆö **exactly symmetric to `AbilityCost::Tap`ŌåÆ`requires`**. Without this rule the `{Q}`-untap-cost engine class produces nothing (its pump/buff effect is `Unmodeled` in PR-4a), so no producer edge forms and the canonical Priest of Titania + Umbral Mantle SCC never closes. This rule is correct regardless of any single card and covers the whole `{Q}`-untap-cost class (Umbral Mantle, Sword of the Paruns, Staff of Domination, Magewright's Stone, Kiora's Follower). It is a leaf arm of the exhaustive no-wildcard `AbilityCost` cost-fold match (┬¦3.3/┬¦4): a new `AbilityCost` variant is a compile error until classified. **Cycle trace (measured AST, ┬¦6 D ŌĆö using the COLLAPSED `AxisKey::Mana`, R3-MANA-COLLAPSE):** Priest of Titania's `{T}: add {G}┬Ę(#Elves)` node produces **green** mana, which under the collapse keys to the single `produces = {AxisKey::Mana}` (Unbounded) + `requires = {AxisKey::Tap}` (its `{T}` cost); the Umbral-Mantle-granted `{3}, {Q}: +2/+2` node has `produces = {AxisKey::Tap}` (its `{Q}` untap cost, this rule) + `requires = {AxisKey::Mana}` (its **generic** `{3}` cost, also collapsed to the one `Mana` key). Both edges therefore form: MantleŌåÆPriest via `AxisKey::Tap` and PriestŌåÆMantle via `AxisKey::Mana` ŌćÆ Tarjan SCC `{Priest, Umbral Mantle}` ŌćÆ coverable ŌćÆ **one** candidate naming `Mana`. Summed mana net is a fixed ŌłÆ3, but the production axis is `Unbounded` ŌćÆ the coverability override (HIGH-1) keeps it. **The collapse is load-bearing here:** Priest produces *green* but Umbral's `{3}` is a *generic/colorless* cost ŌĆö under a per-color `Mana(ManaType)` key the PriestŌåÆMantle edge would NOT form (green production vs colorless-keyed generic cost), the SCC would not close, and the canonical combo would be a false-negative. The off-color over-approximation is filtered downstream by Engine-A's exact color accounting.
