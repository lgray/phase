## 0. Problem restatement (measured, anchored)

The live win-shortcut (`game/engine.rs` `reconcile_terminal_result` ¬ß3, :289‚Äì358, OFF-gated) fires only when the loop-detect ring confirms a repeating board. Ring confirmation is `loop_states_equal_modulo_resources` (`analysis/resource.rs:627`), which projects out monotone resources and canonicalizes stack entry-ids to **positions** but still compares stack **DEPTH exactly** (:817‚Äì819).

- **2-player all-mandatory drain** ‚Äî one drain trigger per cycle, **constant-depth** stack (one in-loop trigger held at every priority window). The state recurs, `loop_states_equal_modulo_resources` matches, `live_mandatory_loop_winner` (`loop_check.rs:212`) names the winner. **Works today.**
- **‚â•3-player all-opponent drain** ‚Äî each cycle the reaction trigger fans out **one trigger per opponent** off distinct life-loss events. With ‚â•2 opponents each generation reproduces more triggers than it consumes ‚áí the **stack place grows without bound** ‚áí successive ring frames have strictly increasing depth ‚áí `loop_states_equal_modulo_resources` never matches ‚áí ring never confirms ‚áí `live_mandatory_loop_winner` is never reached. Even if reached, it hard-gates 2p (`living.len() != 2`, :225) and rejects multi-fallers (:250). **Fails to fire.**

Two independent blockers, both must be lifted:
- **B-STRUCT (the crux):** the exact-depth board fingerprint cannot recognize an unbounded-but-repeating cascade.
- **B-WINNER:** the winner predicate is hard-scoped to 2p / single-faller.

Plus a **prerequisite enabler (PR-6.25):** the fan-out produces simultaneous multi-trigger windows; if they `OrderTriggers`-prompt, the ¬ß2 sampler clears the ring (`engine.rs:632`) and the cascade is never sampled. They must auto-resolve.

## 1. Literature synthesis (verified citations ‚Üí concrete design decisions)

The growing trigger cascade is a **monotone (well-structured) infinite-state transition system**: the stack is a set of counter-"places", each mandatory trigger is a transition that consumes the top entry and produces successor entries, and resolution of the top entry is enabled regardless of how deep the stack is. This is exactly the setting where you **accelerate over the loop symbolically instead of iterating it**.

### 1.1 Karp‚ÄìMiller (1969) coverability tree / œâ-acceleration ‚Äî THE core algorithm
R. M. Karp, R. E. Miller, "Parallel Program Schemata", *J. Comput. Syst. Sci.* 3(2):147‚Äì195, May 1969. Result relied on: for a vector addition system / Petri net, unfolding the reachability graph and, whenever a reachable marking `m'` **covers** an ancestor `m` on the same path (`m' ‚™∞ m` componentwise with `m' ‚âÝ m`), replacing every strictly-grown coordinate with `œâ`, yields a **finite** coverability tree that decides *which places are unbounded* and *whether a marking is coverable* ‚Äî **without enumerating the infinite reachable set**. Modern restatement + termination/soundness: A. Finkel et al., "Commodification of accelerations for the Karp and Miller construction", *Discrete Event Dyn. Syst.* 31 (2021), https://inria.hal.science/hal-03137054v1/document ; formalization: https://dl.acm.org/doi/10.1145/3018610.3018626.
**‚Üí Design decision:** the confirmation gate for a growing cascade is a **covering pair** along the ring: an older ring frame `prior` and the live frame `current` whose normalized stack embeds `prior`'s (order-preserving bottom-up subsequence, ¬ß2.2) with the board equal modulo resources. The strictly-grown stack-entry kind is the œâ place. We never iterate the cascade to completion ‚Äî one covering pair discharges the win (CR 732.2a "a loop that repeats a specified number of times").
**Honesty note (R1-B1/B2):** this is Karp‚ÄìMiller-*inspired*, not Karp‚ÄìMiller verbatim, and the plan says so on the annotation. Classical K‚ÄìM introduces œâ on ANY strictly-grown coordinate ‚Äî **including a place going 0‚Üí1** ‚Äî and that is sound for Petri nets only because transitions are state-independent vector additions ("by repeating the sequence of transitions between the markings one could reach markings with arbitrarily many tokens", Finkel et al. 2021, https://hal.science/hal-03137054/). MTG trigger resolutions are NOT fixed vector additions (a quantity ref or condition can read game state), so we impose two strictly stronger side conditions the classical construction does not need: (1) **prior-occupancy** ‚Äî œâ only on places already occupied in `prior` (count ‚â• 1), so a never-before-seen entry kind (e.g. a conditional board wipe that armed mid-window and sits unresolved on top) can never ride a covering pair to a false win; (2) **order-preserving embedding** ‚Äî `prior`'s normalized stack must be a bottom-up subsequence of `current`'s, so covered entries are the *frozen* lower stack, not a permutation (¬ß2.2 has the LIFO argument). Plus the two-surface resource-read guard of ¬ß1.2/¬ß2.2 items 4‚Äì5, which restores the "transitions behave identically when repeated" premise syntactically over the whole transition relation (on-stack resolution AND off-stack fire-time gating).

### 1.3 Galton‚ÄìWatson / branching-process criticality ‚Äî the "is it growing" test
Extinction‚Äìexplosion dichotomy: a Galton‚ÄìWatson process with mean offspring `Œº` is sub-critical (`Œº<1`, extinction a.s.), critical (`Œº=1`), or super-critical (`Œº>1`, positive explosion probability). Ref: https://en.wikipedia.org/wiki/Galton%E2%80%93Watson_process ; extinction certain iff `Œº‚â§1` (P. Braunsteins et al., *J. Stat. Plan. Inference*, 2016, https://www.sciencedirect.com/science/article/abs/pii/S0378375815000932).
**‚Üí Design decision:** the per-generation fan-out of an N-opponent all-opponent drain has mean offspring `Œº = N` (one reaction trigger per opponent) `> 1` for N‚â•2 ‚áí super-critical ‚áí the stack place is unbounded. We do **not** compute `Œº` numerically; the **covering pair** (¬ß1.1) is the *observed-runtime witness* of `Œº>1` (the place actually grew between two board-equal frames), which is stronger than an estimate and immune to over-approximation. Branching theory is the *why* the cascade is unbounded; coverability is the *decidable certificate*.

### 1.4 Dijkstra‚ÄìScholten (1980) diffusing computations ‚Äî the dual (non-termination)
E. W. Dijkstra, C. S. Scholten, "Termination detection for diffusing computations", *Inf. Process. Lett.* 11(1):1‚Äì4, 1980, DOI 10.1016/0020-0190(80)90021-6 (EWD687a: https://www.cs.utexas.edu/~EWD/transcriptions/EWD06xx/EWD687a.html). Result relied on: a diffusing computation is a graph of message-spawning processes; termination is detected when the spawning deficit balances. Its **dual** ‚Äî the spawn graph never balances because each node spawns ‚â•1 successor ‚Äî is provable non-termination.
**‚Üí Design decision:** the trigger fan-out **is** a diffusing computation whose deficit strictly grows; the covering-pair witness is exactly "the deficit did not shrink between two board-equal frames." Confirms the reconcile-seam `no_living_player_has_meaningful_priority_action` gate (`engine.rs:668`) is the right complement: no external message (player action) can drain the computation.

### 1.5 Motter‚ÄìLai (2002) load-redistribution cascades ‚Äî the framing
A. E. Motter, Y.-C. Lai, "Cascade-based attacks on complex networks", *Phys. Rev. E* 66:065102(R), 2002, https://link.aps.org/doi/10.1103/PhysRevE.66.065102.
**‚Üí Design decision (framing only):** per-cycle fan-out = load propagating through a dependency graph; cascade *size* is unbounded but the *per-node net drain* is what determines the terminal state. Motivates keying the certificate on the **per-opponent net-progress vector** (`ResourceVector.life` restricted to opponents), not on cascade size. No algorithm imported from this thread ‚Äî it is the intuition that the win is decided by the drain, not the branching.

**Synthesis (the certificate):** a growing cascade wins iff there exists a ring frame `prior` such that (a) `current` **covers** `prior` on the stack place ‚Äî order-preserving bottom-up embedding + strict growth confined to places already occupied in `prior` ‚Äî with board equal modulo the NARROWED projection AND the object resource axes strict-equal (`object_resource_axes_match`, R5-B1) [Karp‚ÄìMiller-style œâ-acceleration under the ¬ß1.2 invariance premise], (b) every grown place is a **mandatory, no-ordering-input triggered ability** [the transition that keeps `Œº>1`], (b‚Ä≤) **no still-projected field ‚Äî player-level monotone resource OR journal, the ¬ß1.2(B) closed set ‚Äî is read by any entry on `current`'s stack NOR by any live trigger definition's fire-time condition, replacement definition, or condition-gated static** [the two-surface fail-closed guard that, with the strict compare, makes the ¬ß1.2 premise hold by construction over the whole beat, ¬ß2.2 items 4‚Äì5], (c) the per-cycle `ResourceVector` delta shows the controller net non-losing ‚Äî and never dipping across the window's per-resolution frames (¬ß2.3 m9) ‚Äî and **every** other living player strictly life-losing, (c‚Ä≤) **when there are ‚â•2 fallers, all fallers pairwise life-equal at every ring frame with equal per-cycle deltas** [the R5-B2 simultaneity floor: they cross lethal in ONE CR 704.3 SBA batch, so the first CR 800.4a elimination is the terminal event and nothing past it is modeled], and (d) no living player has a loop-breaking action [CR 732.5]. Then every opponent crosses 0 in the SAME resolution's SBA check after finitely many computable cycles and the controller is the sole survivor ‚Äî CR 104.2a win ("happens immediately"), discharged without iterating.

## 6. Soundness / risk section

**Why the detector never false-positives:**
- `cover_modulo_growth` requires **full board equality modulo the NARROWED projection** (permanents, zones, tap, attachments, phase, priority, per-turn/per-game activation gates, loyalty-activation counts) PLUS **strict equality of the object resource axes** (`damage_marked` + all counters + derived P/T/loyalty/defense, R5-B1) ‚Äî so the CR 704.5f/g/i SBA reads that are part of every beat can never observe hidden object drift (N1(m) pins it).
